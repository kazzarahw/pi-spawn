# pi-spawn — Build Spec

> Design/implementation spec. Decisions, responsibilities, interfaces, build
> order, acceptance criteria. No function bodies.

**Goal:** Delegate tasks to specialized subagents that run in isolated context, with streaming progress, parallel execution, abort, and your own default agent roster.

**Architecture:** Agent definitions discovered from markdown files; a runner that executes one delegated task in an isolated `pi` subprocess and streams its events; a bounded-concurrency pool for parallel delegation; thin Pi glue (tools, streaming widget, config). Composes with pi-git worktrees for isolated parallel filesystem work.

**Tech Stack:** TypeScript/ESM, Bun + `bun test`, `node:child_process` to spawn `pi` in non-interactive JSON mode, `@earendil-works/pi-*` peers.

## Global Constraints

See `pi-shared/HOUSE-STYLE.md`. Application:
- **One tool** `spawn({ tasks })` — a list of jobs; one element is the common single-delegation case, more than one runs in parallel under a concurrency cap. (Merging the old `spawn_parallel` in: a batch has to be a single tool call anyway to get pooled concurrency, so two tools were redundant.)
- Emits `spawn:started { agent }`, `spawn:finished { agent, summary }`.
- Agent-invoked, no automatic behavior → exempt from the enforcement dial (like pi-consult). Command `/pi-spawn` (config). Log `[pi-spawn]`.

## Design Decisions

1. **Isolated `pi` subprocess per subagent** (mirrors the bundled `subagent/` example): a fresh `pi` process with a delegated system prompt, model, and tool set, so the subagent's context never pollutes the parent's. **Confirmed invocation:** `pi --mode json -p --no-session [--model <m>] [--tools <csv>] [--append-system-prompt <tmpfile>] "Task: <task>"` — the system prompt goes to a 0600 temp file; JSON-mode streams JSONL events (`session` header, then `agent_*`/`turn_*`/`message_*`/`tool_execution_*`). Parse `message_end` assistant events for final text + usage. Resolve the `pi` binary via `process.argv[1]`/`process.execPath` (bunfs-virtual → fall back to `"pi"`). Injectable `spawn` fn so tests never launch a real `pi`.
2. **Agent definitions = markdown files** with frontmatter (`name`, `description`, `model?`, `tools?`, body = system prompt). Discover from `.pi/agents/*.md` (project) and `~/.pi/agent/agents/*.md` (global). Ship a **default roster** as bundled `agents/*.md` — *your own* set, not Claude Code's.
3. **v1 tool surface: one `spawn` tool.** `spawn({ tasks: [{ agent, task }], worktree? })` runs each job in an isolated subprocess, streaming progress to a widget; a single-element list is the common case (returns that job's final output), multiple run under a concurrency cap (returns aggregated results). One call blocks until all finish. Chain/async/resume are deferred (the example has them; not v1).
4. **Streaming** — parse subprocess JSON events into concise widget lines (`ctx.ui.setWidget`) per running agent; propagate `ctx.signal` to kill subprocesses on abort.
5. **Worktree isolation (integration) — DEFERRED to v1.1.** The intent: `spawn` requests an isolated git worktree per job so parallel *write* agents don't collide. But pi-git and pi-spawn are separate extensions in the same process with no shared function references, so the seam needs a defined IPC (a `pi.events` request/response, or pi-spawn shelling out to `git worktree` itself). Rather than build that now, v1 runs all jobs in the parent cwd — fine for the default roster (scout/reviewer are read-only; only parallel *worker* jobs risk collision). The `worktree` tool param is dropped from v1; document the collision caveat. Revisit with the IPC seam.

### Open items for the human
- The **default agent roster** (names + roles + prompts) — your call. Placeholder set: `scout` (fast read-only recon → compressed context), `worker` (full capability implementer), `reviewer` (critique). Add/rename freely.
- Default subagent model; parallel concurrency cap (recommend `min(4, cores)`).
- Whether worktree-per-parallel-job is on by default (recommend: off; opt-in flag).

## Interfaces / Contracts

```typescript
interface AgentDef { name: string; description: string; model?: string; tools?: string[]; systemPrompt: string; }
interface SpawnResult { agent: string; ok: boolean; output: string; usage?: { turns: number; tokens: number; cost: number }; }
interface SpawnEvent { kind: "tool" | "text" | "final" | "error"; text: string; }

// agents.ts
function discoverAgents(cwd: string): AgentDef[];        // project ∪ global ∪ defaults, project wins
function defaultAgents(): AgentDef[];                    // parsed from bundled agents/*.md

// runner.ts — spawn injectable
type SpawnFn = (argv: string[], opts: { signal?: AbortSignal; onLine: (line: string) => void }) => Promise<number>;
function runAgent(
  input: { agentDef: AgentDef; task: string; signal?: AbortSignal; onEvent?: (e: SpawnEvent) => void },
  spawn?: SpawnFn,
): Promise<SpawnResult>;

// pool.ts
function runParallel(
  jobs: Array<{ agentDef: AgentDef; task: string }>,
  concurrency: number,
  run?: typeof runAgent,
  signal?: AbortSignal,
): Promise<SpawnResult[]>;

// render.ts (pure)
function eventToLine(e: SpawnEvent): string;

// tool: spawn({ tasks: [{ agent, task }] })  — 1 job or many; parallel + pooled when >1 (worktree deferred, Decision 5)
// events: spawn:started {agent}, spawn:finished {agent, summary}
// fork-bomb guard: refuse when PI_SPAWN_DEPTH >= 2; children inherit PI_SPAWN_DEPTH+1
```

## Build Sequence

- [ ] **1 · Scaffold** — per house-style, plus a bundled `agents/` dir. Accept: `bun test` runs.
- [ ] **2 · Agent discovery** — Accept: parses frontmatter+body from a fixture `.md`; `discoverAgents` merges project/global/defaults with project winning on name collisions; malformed files are skipped with a logged warning.
- [ ] **3 · Runner (fake spawn)** — Accept: builds the correct `pi` argv (JSON mode, model, delegated prompt) for an `AgentDef`+task; parses a fed JSON event stream into `SpawnResult` (`ok`, `output`, `usage`); a non-zero exit → `ok:false`. No real `pi` in tests.
- [ ] **4 · Pool** — Accept: `runParallel` never exceeds `concurrency` in flight (assert via a fake run that records overlap); aborting the signal cancels pending + in-flight jobs; results preserve input order.
- [ ] **5 · Render** — Accept: `eventToLine` maps each `SpawnEvent.kind` to a concise, prefixed line.
- [ ] **6 · Tool** — Accept: `spawn` with a single-job `tasks` list resolves the agent by name, runs it, emits `spawn:started`/`spawn:finished`, returns final output; an unknown agent name → thrown tool error listing available agents; a multi-job `tasks` list runs under the concurrency cap and returns aggregated results in input order.
- [ ] **7 · Default roster + config** — Accept: bundled `agents/*.md` load as defaults; `/pi-spawn` sets default model/agent-dir/concurrency; manual smoke: delegate a real recon task to `scout` and see streamed progress + a returned summary.

## Test Strategy

Pure/isolated units carry it: agent parsing, `eventToLine`, argv construction, event-stream parsing, and pool concurrency/abort — all with a fake `SpawnFn` that emits canned event lines and records concurrency. One manual smoke spawns a real `pi` subagent.

## Risks / Notes

- Depends on Pi's JSON/RPC subprocess contract (`json.md`) — pin the exact event shapes during step 3; if the schema differs, only the parser changes.
- Recursion/fork-bomb risk (**handled**): a subagent that loads pi-spawn could spawn endlessly. Guard: children inherit `PI_SPAWN_DEPTH+1` in their env; the tool refuses when depth ≥ `MAX_DEPTH` (2). Subagents still load the full extension suite (no `--no-extensions`) so they can use pi-lens/pi-git etc.; the depth guard is what bounds nesting.
- Cost/latency of parallel real subagents is real — surface usage in the widget and keep the concurrency cap conservative.
- Worktree integration couples to pi-git; keep it behind a flag so pi-spawn works standalone.
